Accepted solution

One reason to get this error is if the Add-In's manifest file doesn't have an id field defined.

 

This can easily happen for two reasons:

    The documentation explicitly says that the id field is not needed: "This property is not used and can be left empty."
    When creating a new add-in, the default template fusion generates does not set an id.

 

Fusion has to be restarted for a change to the manifest file to take effect. Just starting/stopping the add-in is not sufficient.

 

Example manifest with an id (a randomly generated UUID works fine):

 

{
	"autodeskProduct":	"Fusion",
	"type":	"addin",
	"id": "8ddf023b-89bd-4313-84ea-1d94c4dba3bd",
	"author":	"...",
	"description":	{
		"":	""
	},
	"version":	"",
	"runOnStartup":	false,
	"supportedOS":	"windows|mac",
	"editEnabled":	true,
	"iconFilename":	"AddInIcon.svg"
}

 

It took me days to figure this out. I hope this helps someone. Autodesk, a more descriptive error message would be much appreciated.

Glad it was helpful to you.

 

After a frustrating day or two, I decided to approach the problem slowly but systematically.

 

I simplified my add-on to the bare minimum needed to get to the failing function call and it still wouldn't work.

 

Then I took a working add-in from the app store that uses this feature as a reference. I started deleting code in it bit by bit, reducing it too to a bare minimum example.

 

I figured as the two add-ins get increasingly similar, the reference add-in would eventually produce this error too, and then I'd know that the last bit of code that I removed/changed from the reference add-in is what my add-in is missing.

 

In the end, I had both add-ins to the point where their python code was 100% identical. And yet one add-in produced this error and the other didn't. I copied the files back and forth several times. I just couldn't believe it.

 

The only explanation left was that the difference had to be in a non-python file. That's when I started comparing the manifest. 

 

I can see why fusion needs the id. When storing/loading a design from/to cloud or disk, it needs to somehow associate the feature in the design's timeline with the add-in. It needs an identifier that won't change and will be identical on all installations.
